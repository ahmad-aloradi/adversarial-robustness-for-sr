<objective>
Fix MagnitudePruner to properly suppress validation during sparsity ramp-up, and write verification tests proving the fix works.

Purpose: The current implementation manipulates EarlyStopping/ModelCheckpoint state but doesn't prevent validation from actually executing. This wastes compute and produces misleading "New best score" log messages at low sparsity. The fix uses `trainer.limit_val_batches = 0` to skip validation entirely during ramp-up.

Output: Fixed `prune.py` with proper validation suppression + comprehensive test file.
</objective>

  <action>
Modify `MagnitudePruner` to use `trainer.limit_val_batches = 0` for proper validation suppression during sparsity ramp-up. The key changes:

1. **Add `_original_limit_val_batches` to `__init__`**: Initialize as `None`. This stores the trainer's original `limit_val_batches` value.

2. **Modify `on_train_epoch_start`**: After the existing pruning logic, add validation control:
   - Compute whether target sparsity is reached: `target_reached = current_sparsity >= (self.final_amount - 1e-4)` (use `new_sparsity` after pruning is applied, which is already computed in the method).
   - If `self.scheduled` and not `target_reached`:
     - On first call, save `trainer.limit_val_batches` to `self._original_limit_val_batches` (only save once, when it's still `None`)
     - Set `trainer.limit_val_batches = 0`
   - If `self.scheduled` and `target_reached` and `self._original_limit_val_batches is not None`:
     - Restore `trainer.limit_val_batches = self._original_limit_val_batches`
     - Set `self._original_limit_val_batches = None` (so we don't restore twice)
     - Log: `"Target sparsity reached. Restoring validation."`

3. **Simplify `_manage_metric_trackers`**: The existing tracker management in `on_train_epoch_end` should STILL reset EarlyStopping/ModelCheckpoint state when validation resumes (the "ENABLE" branch). Keep the `save_top_k=0` logic as a secondary safety net. But validation skipping is now the primary mechanism.

4. **Do NOT remove the existing `_manage_metric_trackers` method** -- keep it as-is for the state reset functionality when transitioning from ramp to post-ramp. The `limit_val_batches=0` prevents validation from running; the tracker reset ensures clean state when validation resumes.

Important constraints:
- Do NOT use `on_validation_epoch_start` hook override (this fights the framework)
- Do NOT use `trainer.should_stop` (nuclear option)
- The fix must work with Lightning 2.x callback system
- Preserve all existing checkpoint save/load logic
