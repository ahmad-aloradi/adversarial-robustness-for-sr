<objective>
Implement a "relaxation mode" for Bregman learning where the LambdaScheduler's target_sparsity follows a schedule instead of being fixed.

Purpose: In standard Bregman learning, the model starts maximally sparse and lambda drives toward a fixed target. In the relaxation mode, the target itself evolves: starting at the initial sparsity (e.g., 0.99) and gradually relaxing downward to the final target (e.g., 0.9) over N epochs. This gives the model more capacity gradually, mirroring how magnitude pruning ramps sparsity. Lambda only *indirectly* affects sparsity, so the schedule modifies `target_sparsity` over time and lambda chases that moving target.

Output: Modified LambdaScheduler and BregmanPruner with scheduled target support, new experiment config, and verification tests.
</objective>

**A) LambdaScheduler changes** (`src/callbacks/pruning/bregman/lambda_scheduler.py`):

Add optional schedule parameters to `__init__`:
- `schedule_type: str | None = None` -- None means fixed target (current behavior), "linear" or "constant" enables schedule
- `initial_target_sparsity: float | None = None` -- starting target (high, e.g. 0.99, matching model's initial sparsity)
- `final_target_sparsity: float | None = None` -- ending target (lower, e.g. 0.9, the desired operational sparsity)
- `epochs_to_ramp: int = 10` -- number of epochs for the schedule to complete

When `schedule_type is not None`:
- Validate that `initial_target_sparsity` and `final_target_sparsity` are provided and in (0.0, 1.0]
- Set `self.target_sparsity = initial_target_sparsity` (overriding the `target_sparsity` init param -- or use `initial_target_sparsity` as the starting value, ignoring `target_sparsity`). The cleanest approach: when schedule is active, `target_sparsity` param is ignored, and `self.target_sparsity` is set to `initial_target_sparsity`.
- Store `self._schedule_type`, `self._initial_target_sparsity`, `self._final_target_sparsity`, `self._epochs_to_ramp`
- Add internal `self._schedule_epoch: int = 0` to track progress

Add method `update_target(self, current_epoch: int) -> float`:
- If `self._schedule_type is None`, return `self.target_sparsity` unchanged (no-op for fixed mode)
- Compute `progress = min(1.0, (current_epoch + 1) / self._epochs_to_ramp)`
- Use the SAME interpolation math as `PruningScheduler._generate_schedule()` (see `src/callbacks/pruning/scheduler.py`):
  - For "linear": `target = initial + (final - initial) * progress`
    (where initial = `self._initial_target_sparsity`, final = `self._final_target_sparsity`)
  - For "constant" (constant pruning rate -- log-space interpolation):
    ```python
    remaining_final = 1.0 - self._final_target_sparsity
    remaining_initial = 1.0 - self._initial_target_sparsity

    if remaining_final <= 1e-9:
        remaining_final = 1e-9
    if remaining_initial <= 1e-9:
        target = 1.0
    else:
        # Log-space interpolation: same formula as PruningScheduler
        current_remaining = remaining_initial * (remaining_final / remaining_initial) ** progress
        target = 1.0 - current_remaining
    ```
    Then clamp: `target = min(max(target, 0.0), max(self._initial_target_sparsity, self._final_target_sparsity))`
    NOTE: In the Bregman context initial > final (relaxing downward), but the math is identical to PruningScheduler. The key formula is `current_remaining = remaining_initial * (remaining_final / remaining_initial) ** progress` -- this produces log-space interpolation between the two remaining-weight fractions.
- Set `self.target_sparsity = target`
- Store `self._schedule_epoch = current_epoch`
- Return the new target value
- Log at info level: `f"Target sparsity updated: {self.target_sparsity:.4f} (epoch {current_epoch}, schedule progress {progress:.1%})"`

Add property `is_scheduled(self) -> bool`: return `self._schedule_type is not None`

Add property `schedule_complete(self) -> bool`: return `not self.is_scheduled or self._schedule_epoch >= self._epochs_to_ramp - 1`

Update `get_state()` to include schedule fields: `_schedule_type`, `_initial_target_sparsity`, `_final_target_sparsity`, `_epochs_to_ramp`, `_schedule_epoch`. Use `.get()` defaults in `load_state()` for backward compatibility with old checkpoints that lack these fields (default to None/0).

Update `load_state()` to restore schedule fields. If checkpoint has `_schedule_type`, restore all schedule fields. If not, set `_schedule_type = None` (fixed mode, backward compatible).

**B) BregmanPruner changes** (`src/callbacks/pruning/bregman/bregman_pruner.py`):

Add `self._original_limit_val_batches: Optional[int] = None` to `__init__` (same pattern as MagnitudePruner from 04-01).

Add `on_train_epoch_start` hook (new method):
```python
def on_train_epoch_start(self, trainer: Trainer, pl_module: LightningModule) -> None:
    if not self._initialized or self.lambda_scheduler is None:
        return

    # Update scheduled target (no-op if fixed mode)
    if self.lambda_scheduler.is_scheduled:
        new_target = self.lambda_scheduler.update_target(trainer.current_epoch)

        # Validation suppression during schedule ramp
        if not self.lambda_scheduler.schedule_complete:
            # Save original limit_val_batches ONCE on first ramp epoch
            if self._original_limit_val_batches is None:
                self._original_limit_val_batches = trainer.limit_val_batches
            trainer.limit_val_batches = 0
        elif self._original_limit_val_batches is not None:
            # Guard: only restore ONCE, then clear the saved value
            # This elif prevents repeated restoration on subsequent epochs
            # after the schedule completes (because _original_limit_val_batches
            # is set to None after the first restoration).
            trainer.limit_val_batches = self._original_limit_val_batches
            self._original_limit_val_batches = None
            log.info("Bregman target schedule complete. Restoring validation.")
```

The guard logic works as follows:
- During ramp (schedule_complete=False): saves original `limit_val_batches` on first entry, sets to 0 each epoch
- On first epoch after ramp completes (schedule_complete=True AND `_original_limit_val_batches is not None`): restores the original value and clears the saved value
- On all subsequent epochs (schedule_complete=True AND `_original_limit_val_batches is None`): the `elif` condition is False, so no restoration attempt -- this prevents repeated/spurious restoration

Update `on_save_checkpoint` to include schedule state (already handled by `lambda_scheduler.get_state()` -- no change needed here, just verify).

Update `_log_configuration` to log schedule info when active (optional, nice-to-have):
- If `self.lambda_scheduler.is_scheduled`, log schedule_type, initial_target, final_target, epochs_to_ramp.
